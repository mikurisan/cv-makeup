# 什么是多模态原始数据?

在你这个"具身机器人"项目里, 多模态原始数据指的是: **机器人在遥操作(遥操)采集过程中, 同一时刻由不同种类传感器同步记录下来的/未经处理的多种异构数据流**.

## 拆解三个关键词

**"多模态" (Multi-modal)**

"模态"就是数据的种类/来源. 人感知世界靠视觉 / 听觉 / 触觉, 机器人也一样, 靠不同传感器获取不同维度的信息. 每一种传感器数据就是一个"模态".

**"原始" (Raw)**
指未经清洗 / 对齐 / 格式转换的数据. 直接从传感器/ROS 话题(topic)里录制下来的样子, 通常就是 ROS bag 文件. 这也是你项目里 ETL 流水线要处理的"输入".

**"数据" (Data)**
承载信息的实体, 在ROS 世界里就是一条带时间戳的消息(message).

## 具身机器人里典型的模态有哪些

结合遥操场景, 常见的模态大致是这几类:

| 模态类型 | 典型传感器 / 数据 | ROS 消息类型举例 | 特点 |
|---------|----------------|------|------|
| [视觉](./ca.md) | RGB 相机 / 深度相机(RGB-D) / 多路相机(手腕/头部/第三视角) | `sensor_msgs/Image`, `CompressedImage` | 数据量大, 帧率高(30Hz) |
| 深度/点云 | 深度相机 / 激光雷达 | `sensor_msgs/PointCloud2` | 稀疏, 数据大 |
| 本体状态 | 关节编码器 / 机械臂/机器人本体反馈 | `sensor_msgs/JointState` | 高频(100~1000Hz) |
| 动作指令 | 遥操设备(主手/手柄/VR)下发的控制指令 | `geometry_msgs/Twist`, 自定义 action msg | 训练时是"标签" |
| 力/触觉 | 力矩传感器 / 触觉传感器 | `geometry_msgs/WrenchStamped` | 高频 |
| 位姿 | IMU / 末端执行器位姿 / TF 变换 | `sensor_msgs/Imu`, `tf2_msgs/TFMessage` | 高频 |
| 夹爪状态 | 夹爪开合 / 抓取力 | 自定义 msg | 低频 |

在模仿学习(Imitation Learning)/具身智能的语境下, 这些数据最终要组织成 **(观测 observation, 动作 action)** 的配对样本, 喂给策略模型训练. 视觉+本体状态通常是"观测", 遥操指令是"动作/标签".

## 为什么"多模态"是你项目的技术难点

这一点很重要, 面试官大概率会顺着问. 多模态数据难就难在**异构性**:

- **频率不同(异频)**: 相机 30Hz, 关节状态 1000Hz, 夹爪 10Hz. 你简历里写的"多传感器异频时间戳对齐"正是在解决这个问题.
- **数据量差异大**: 一路 1080p 图像流一分钟就上 GB, 而 JointState 可能才几 MB. 这直接影响你为什么选 MinIO 存原始大文件 / MongoDB 只存元信息索引.
- **格式各异**: 图像是二进制帧, 点云是结构化数组, 状态是浮点向量. ETL 要把它们统一裁剪、重映射, 最终转成 LeRobot/HDF5 这种规整的训练格式.

一句话总结, 你可以在面试里这么说:

> "多模态原始数据就是机器人遥操采集时, 各类传感器(多路相机/深度/关节状态/力觉/遥操动作指令等)同步产生的, 未经处理的异构数据流, 通常以 ROS bag 形式落盘. 它们频率不同 / 数据量差异大 / 格式各异, 我做的 ETL 流水线核心工作之一就是把这些异频异构的数据做时间戳对齐并转换成规整的训练格式."
