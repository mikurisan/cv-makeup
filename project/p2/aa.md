# 什么是块存储?

块存储 (Block Storage) 是最底层, 最接近物理硬件的一种存储范式. 它把存储空间切分成固定大小的"块 (Block)", 每个块有自己的地址, 上层系统可以像操作一块**裸硬盘**一样直接读写这些块.

## 核心理解: 它给你的是"生磁盘"

块存储不关心你存的是文件, 数据库还是别的什么, 它只提供一块"生的, 未格式化的裸磁盘". 你拿到之后需要自己:

1. **分区** (partition)
2. **格式化文件系统** (ext4 / xfs / NTFS 等)
3. **挂载** (mount) 到操作系统上使用

挂载完成后, 对操作系统来说它就是一块普通的本地磁盘, 比如 Linux 下的 `/dev/sdb`, 或者云上挂载的一块云盘.

## 三种存储范式的层级关系

这里最容易混淆, 用一个比喻理清:

```
文件存储  →  像"精装修的房子", 拎包入住 (有目录、有文件系统, 直接放东西)
块存储    →  像"毛坯房", 给你空间, 装修 (格式化) 后才能住
对象存储  →  像"仓库托管", 你把整箱货 (对象) 交给仓库, 给你一个取货单号 (key)
```

实际上, **文件存储和对象存储的底层往就是构建在块存储之上的**. 块存储是地基.

## 典型特征

| 维度 | 块存储的表现 |
|------|------|
| 访问粒度 | 块 (通常 512B ~ 4KB), 可随机读写任意位置 |
| 访问协议 | iSCSI、FC (光纤通道)、NVMe-oF, 云上则是挂载云盘 |
| 是否可局部修改 | **可以**, 改文件中间一小段不需要重写整个文件 |
| 性能 | 延迟最低、IOPS 最高 |
| 共享性 | 通常一块盘只挂载给一个实例 (独占) |

## 最关键的一点: 支持随机读写

这是块存储和对象存储最本质的区别, 也是面试高频考点.

- **块存储**: 可以精确修改某个块. 比如一个 100GB 的数据库文件, 我只改其中 4KB, 底层只写那 4KB 对应的块. 这对数据库至关重要.
- **对象存储 (如 MinIO)**: 对象是"不可变 (immutable)"的整体. 想改一个对象里的一个字节, 本质上要**重新上传整个对象**. 所以它不适合频繁小改动的场景.

这就是为什么:
- **数据库、虚拟机磁盘 → 用块存储** (需要频繁随机读写)
- **海量原始数据、备份、日志 → 用对象存储** (一次写入, 多次读取, 很少修改)

## 回到你的项目

你的项目里其实**没有直接用块存储**, 但理解它能帮你回答一个很可能被追问的问题:

> "你们的 ROS bag 数据为什么选对象存储 MinIO, 而不是块存储?"

参考应答思路:

> "我们的多模态原始数据有几个特点: 单文件大 (GB 级)/一次写入后基本只读不改/数据量持续增长且需要横向扩展. 块存储虽然读写性能最强, 但它是独占挂载/容量受单盘限制, 也不天然支持 HTTP 访问和多维检索. 而对象存储天然适合这种海量/非结构化/写少读多的数据, 还能通过 S3 API 让 ETL 流水线和质检工具直接远程访问. 块存储更适合我们元数据服务 MongoDB 底层的数据盘这种需要随机读写的场景."

注意最后一句是个巧妙的收尾: 它说明你不只是"知道对象存储好", 而是理解了**块存储也有它的位置** (MongoDB 的数据文件跑在块存储/云盘上). 这种"分层认知"正是面试官想看到的深度.
