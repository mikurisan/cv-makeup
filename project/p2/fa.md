# 什么是 SQLite3?

SQLite3 是一个**嵌入式的/轻量级的关系型数据库**. 它最大的特点是: **整个数据库就是一个普通文件, 不需要独立的数据库服务进程**. 你[不用像 MySQL/PostgreSQL 那样先启动一个数据库服务器再去连接, 直接以库(library)的形式嵌入到你的程序里](./faa.md), 读写一个 `.db` / `.db3` 文件就行.

在你的项目语境里: ROS2 bag 默认就是用 SQLite3 来存录制的消息数据, 那个 `.db3` 文件本质上就是一个 SQLite 数据库.

## 核心特点

**嵌入式 (Embedded / Serverless)**

没有独立的服务进程. 数据库引擎是一段库代码, 直接链接进应用程序. 对比一下:

| | 传统数据库 (MySQL/PostgreSQL) | SQLite3 |
|------------------------|------|------|
| 架构 | 独立服务进程, C/S 模式 | 嵌入进程内, 无服务 |
| 部署 | 需要安装、启动、配置 | 一个文件, 零配置 |
| 网络 | 通过 TCP/socket 连接 | 直接读写本地文件 |
| 并发写 | 支持高并发 | **写操作串行(库级锁)** |
| 适用场景 | 多用户/高并发服务端 | 单机、嵌入式、本地存储 |


**单文件存储**
整个数据库(表/索引/数据)都在一个文件里. 拷贝这个文件就等于拷贝整个数据库. 这也是为什么 ROS2 bag 用它——方便打包、迁移一份采集数据.

**零配置/跨平台**
不需要账号密码/不需要配置网络端口, 拿来即用. 这也是它成为世界上部署最广泛的数据库的原因(手机 App/浏览器/操作系统里到处都是它).

**支持标准 SQL 和事务(ACID)**
虽然轻量, 但它是正经的关系型数据库, 支持 SQL 查询 / 事务 / 外键. 数据写入有事务保证, 不会写一半崩掉留下脏数据.

## ROS2 bag 里 SQLite3 长什么样

一个 ROS2 bag 的 `.db3` 文件, 用 SQLite 打开看, 大致是这样的表结构:

```sql
-- topics 表: 记录这个 bag 里有哪些话题
CREATE TABLE topics (
    id INTEGER PRIMARY KEY,
    name TEXT,              -- 话题名, 如 /camera/image_raw
    type TEXT,              -- 消息类型, 如 sensor_msgs/msg/Image
    serialization_format TEXT  -- 序列化格式, ROS2 里是 cdr
);

-- messages 表: 实际的消息数据
CREATE TABLE messages (
    id INTEGER PRIMARY KEY,
    topic_id INTEGER,      -- 外键, 关联到 topics.id
    timestamp INTEGER,     -- 消息时间戳(纳秒)
    data BLOB              -- 序列化后的消息二进制数据(CDR 格式)
);
```

理解这个结构对你很有用: 你 ETL 流水线做"流式解析", 本质上就是从 `messages` 表按 `timestamp` 顺序读出 `data` 这个 BLOB, 再用对应的消息类型反序列化. 你甚至可以直接用 SQL 去查某个时间段 / 某个 topic 的消息, 这在调试时很方便.

## 为什么它在你的高频采集场景下会成为瓶颈

上一轮我们聊到"从 sqlite3 换成 mcap", 根因就在 SQLite3 的设计定位上. 这里把逻辑补全, 面试被追问时你能答到点子上:

**1. 写操作串行 (库级写锁)**
SQLite3 为了保证数据一致性, 同一时刻只允许一个写事务, 整个数据库文件级别加锁. 多路相机 + 高频关节状态同时高速写入时, 写请求排队, 容易成为瓶颈, 极端情况下上游来不及录导致**丢帧**.

**2. 面向随机读的 B-tree 结构, 不是为顺序流式追加优化**
它底层是 B-tree, 适合"根据主键随机查一条". 但 bag 录制是"不停往后追加海量消息"这种顺序写场景, 大文件下 B-tree 的写放大和维护开销会拖慢速度.

**3. [BLOB](./fab.md) 存大二进制数据效率一般**
图像帧/点云这种大 BLOB 塞进关系型数据库, 存储和读取效率都不如专门的流式格式(mcap 有分块 chunk + 压缩).

一句话逻辑链: **SQLite3 是通用型嵌入式关系库, 强在"单机/事务/随机查询", 弱在"高频顺序流式大数据写入"**——而机器人多模态采集恰恰是后者, 所以工业界高吞吐场景更倾向 mcap.

## 你可以怎么用它做加分表述

> "ROS2 bag 默认存储后端 SQLite3 本质是个嵌入式关系库, `.db3` 里就是 topics 和 messages 两张表, 消息以 BLOB 存. 它的优势是零配置 / 单文件 / 支持 SQL, 我们调试时能直接 SQL 查特定 topic 和时间段的数据. 但它写操作是串行的, B-tree 面向随机读, 在我们多路高频采集的顺序流式写场景下会有写瓶颈, 所以生产采集我们切到了 mcap."

顺带一提, 你项目里如果需要一个**轻量的本地元信息缓存/中间态存储**(比如 ETL 处理过程中的临时索引), SQLite3 其实是很好的选择——它和你已经在用的 MongoDB 定位互补: MongoDB 管全局的/可检索的元信息索引, SQLite 适合单机流程内的临时结构化数据. 这也是一个能体现你"选型有分寸感"的细节.
