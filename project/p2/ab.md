# 什么是文件存储?

文件存储 (File Storage) 是我们**最熟悉/最直觉**的一种存储范式. 你电脑里的 C 盘/D 盘, 公司里的共享盘, Linux 上的目录, 本质上都是文件存储. 它以"文件"为单位, 通过**层级目录树 (hierarchical directory tree)** 来组织数据.

## 核心理解: 目录 + 文件的树状结构

文件存储的组织方式就是我们天在用的那套:

```
/home
  └── robot-data
        ├── 2024
        │     ├── scene01
        │     │     ├── episode_001.bag
        │     │     └── episode_002.bag
        │     └── scene02
        │           └── episode_003.bag
        └── config.yaml
```

你通过**路径 (path)** 来定位一个文件, 比如 `/home/robot-data/2024/scene01/episode_001.bag`. 注意这里的斜杠是**真实存在的层级**, 每一层都是一个真正的目录节点, 这跟对象存储用斜杠"假装"目录有本质区别 (这一点后面会展开).

## 关键: 遵循 POSIX 语义

文件存储最大的特点是它提供了一套标准的**文件操作接口 (POSIX 语义)**:

- 打开 / 关闭文件 (open / close)
- 读 / 写 (read / write), 支持**从文件任意位置读写** (seek)
- 追加 (append)
- 修改文件属性 (权限/时间戳/所有者)
- 目录遍历 (list directory) / 重命名 / 移动

正因为遵循 POSIX, **应用程序几乎不用做任何适配**就能读写文件存储, 这是它易用性的根源. 你的 Python 代码里一句 `open("/data/xx.bag")` 就能读, 不需要引入任何 SDK.

## 本地文件系统 vs 网络文件存储

"文件存储"这个词其实涵盖两个层次, 面试时区分清楚会显得专业:

| 类型 | 说明 | 例子 |
|------|------| ------|
| 本地文件系统 | 建立在本机块设备上的文件系统 | ext4 / xfs / NTFS |
| 网络文件存储 | 通过网络协议共享给多台机器 | NFS / SMB/CIFS / 云上的 NAS/EFS |

在分布式/多节点场景下, 我们说的"文件存储"通常指**网络文件存储 (如 NFS)**: 多台机器可以同时挂载同一个共享目录, 像访问本地文件一样问远程文件.

## 三种范式再对照一次

```
块存储    →  给你"裸磁盘", 你自己装文件系统 (最底层)
文件存储  →  给你"带目录树的文件系统", 拎包入住 (中间层)
对象存储  →  给你"扁平的桶 + 取货单号 key" (上层, 面向海量数据)
```

关键的依赖关系: **文件存储通常构建在块存储之上**. 块存储提供裸空间, 文件系统 (ext4 等) 在上面组织成目录和文件.

## 文件存储的优势与局限

优势:
- **易用**: POSIX 接口, 应用零改造, 直接读写
- **支持随机读写和局部修改**: 改文件中间一段不用重写整个文件 (对象存储做不到)
- **人类友好**: 目录层级直观, 便于管理

局限 (这是它在海量数据场景被对象存储替代的原因):
- **元数据瓶颈**: 目录树需要维护大量元数据. 当一个目录里有几百万个文件时, 光是 `ls` 遍历就会很慢
- **扩展性差**: 传统文件系统难以横向扩展到 PB 级/百亿文件级
- **共享一致性复杂**: NFS 多节点同时读写同一文件时, 锁和一致性处理很麻烦
- **不天然支持 HTTP/多维检索**: 只能靠路径找文件, 想按"传感器类型 + 场景 + 时间"做多维检索, 文件系统本身无能为力

## 回到你的项目: 一个非常重要的对比点

你的项目里用了 MinIO (对象存储) + MongoDB (元数据索引), 面试官极可能这样追问:

> "你们的 ROS bag 数据不就是一堆文件吗? 为什么不直接用 NFS 这种文件存储放在共享盘上, 反而要引入 MinIO?"

参考应答思路:

> "早期确实可以用 NFS 快速搭起来, 但我们评估后发现几个硬伤: 第一, 数据量会持续增长到 TB 甚至 PB 级, 且文件数量庞大, 文件系统的元数据管理和目录遍历会成为瓶颈, 一个目录下几十万个 episode, `ls` 都会卡; 第二, NFS 横向扩展困难, 而对象存储通过纠删码天然支持分布式扩容; 第三, 也是最关键的, 我们需要按'场景/传感器配置/采集时间'等多个维度做检索, 文件系统只能靠路径线性查找, 而对象存储 + MongoDB 元数据索引的组合, 让我们把'检索'和'存储'解耦, 查询走 MongoDB 的索引, 拿到 key 再去 MinIO 取数据, 检索效率和灵活性都不是文件系统能比的. 另外对象存储的 S3 API 让远程访问和权限控制也更规范."

这个回答的价值在于: 你不是简单说"对象存储更好", 而是**基于数据规模/访问模式/检索需求**给出了有工程判断的选型理由. 面试官想听的正是这种"权衡 (trade-off)"意识.
